# 多级兜底策略

> 本文档描述当 MiniMax 大模型服务不可用时的多级兜底策略，确保系统高可用。

---

## 1. 设计背景

### 1.1 单点故障风险

```
传统架构的问题：
- 依赖单一模型（MiniMax M2.7）
- 模型服务不可用 → 整个系统不可用
- 无降级方案
- 用户体验差
```

### 1.2 解决思路

```
多级兜底设计：
- 第1层：同一厂商降级（M2.7 → M2.5）
- 第2层：不同厂商切换（MiniMax → 豆包/通义）
- 第3层：规则引擎兜底（简单操作走规则）
- 第4层：人工处理（极端情况）
```

## 2. 兜底层级设计

### 2.1 四级兜底架构

```
┌─────────────────────────────────────────────────────────────┐
│                    四级兜底架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 1：MiniMax M2.7（主服务）                          │
│      ↓ 不可用                                              │
│  Level 2：MiniMax M2.5（同一厂商降级）                    │
│      ↓ 不可用                                              │
│  Level 3：豆包/通义（不同厂商）                            │
│      ↓ 不可用                                              │
│  Level 4：规则引擎 + 人工处理                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 兜底触发条件

| 层级 | 触发条件 | 切换时间 | 用户感知 |
|------|----------|----------|----------|
| **Level 1** | 正常服务 | - | 无 |
| **Level 2** | M2.7 响应>10s或错误率>5% | <1s | 无 |
| **Level 3** | M2.5 响应>10s或错误率>5% | <5s | 略有延迟 |
| **Level 4** | 所有模型不可用 | >30s | 需要人工确认 |

### 2.3 兜底时的限制

```
┌─────────────────────────────────────────────────────────────┐
│                    兜底时的操作限制                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 2（M2.5兜底）：                                     │
│  ├── 功能正常                                               │
│  ├── 响应可能变慢                                           │
│  └── 可能需要降级到2票制（如果M2.5理解能力不足）           │
│                                                             │
│  Level 3（豆包/通义兜底）：                                 │
│  ├── L0/L1操作继续可用                                     │
│  ├── L2操作降级为1票（主Agent）                           │
│  └── L3操作提示"服务繁忙，稍后再试"                        │
│                                                             │
│  Level 4（规则引擎兜底）：                                  │
│  ├── 仅支持预定义的标准操作                                 │
│  ├── 非标准操作需要人工处理                                 │
│  └── 限制操作额度                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 模型适配器设计

### 3.1 统一接口

```typescript
interface LLMAdapter {
  // 统一的模型调用接口
  call(prompt: string, context: Context): Promise<Response>;

  // 健康检查
  health(): Promise<boolean>;

  // 获取模型能力
  capabilities(): ModelCapabilities;
}

// 模型注册表
const modelRegistry = {
  'minimax-m2.7': MiniMaxM27Adapter,
  'minimax-m2.5': MiniMaxM25Adapter,
  'doubao': DoubaoAdapter,
  'qwen': QwenAdapter,
};
```

### 3.2 自动切换逻辑

```typescript
class ModelRouter {
  private adapters: LLMAdapter[];
  private currentIndex = 0;

  async call(prompt: string, context: Context): Promise<Response> {
    for (let i = this.currentIndex; i < this.adapters.length; i++) {
      const adapter = this.adapters[i];

      try {
        const result = await this.callWithTimeout(adapter, prompt, context);
        this.currentIndex = i; // 记录成功的层级
        return result;
      } catch (error) {
        if (this.isRetryableError(error)) {
          continue; // 尝试下一个
        }
        throw error;
      }
    }

    throw new AllModelsFailedError();
  }
}
```

## 4. 规则引擎兜底

### 4.1 规则引擎能力

```
┌─────────────────────────────────────────────────────────────┐
│                    规则引擎能力范围                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【支持的操作】                                              │
│  ├── L0查询（固定条件）                                    │
│  ├── L1简单创建（预定义模板）                              │
│  ├── 状态查询                                             │
│  └── 数据统计                                             │
│                                                             │
│  【不支持的操作】                                           │
│  ├── 复杂意图理解                                         │
│  ├── 非标准化操作                                         │
│  ├── 模糊语义解析                                         │
│  └── 跨表关联操作                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 规则引擎降级策略

```markdown
当所有模型不可用时：

L0查询：
→ 使用规则引擎直接查询数据库
→ 返回缓存数据（如果有）

L1创建：
→ 检查是否符合预定义模板
→ 符合则进入"待确认"队列
→ 不符合则返回"需要人工处理"

L2操作：
→ 进入"待确认"队列
→ 通知相关人员
→ 用户选择等待或取消
```

## 7. 监控与告警

### 7.1 关键监控指标

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 模型可用率 | 各层级模型的可用时间比例 | <99.9% |
| 降级次数 | 进入降级状态的次数 | >3次/天 |
| 兜底触发次数 | 各级兜底被触发的次数 | - |
| 恢复时间 | 从降级到恢复的平均时间 | >5分钟 |
| 人工处理工单数 | 需要人工处理的工单数 | >10个/天 |

### 7.2 告警级别

| 级别 | 触发条件 | 通知方式 |
|------|----------|----------|
| **P1** | Level 1不可用 | 立即电话+短信 |
| **P2** | Level 2不可用 | 短信+钉钉 |
| **P3** | 降级到Level 3 | 钉钉通知 |
| **P4** | 降级到Level 4 | 邮件+工单 |

## 8. 成本考虑

### 8.1 兜底成本对比

| 层级 | 模型成本 | 延迟 | 用户体验 |
|------|----------|------|----------|
| Level 1（M2.7） | 基准 | 基准 | 最佳 |
| Level 2（M2.5） | 约60% | +20% | 略差 |
| Level 3（豆包） | 因厂商而异 | +30-50% | 一般 |
| Level 4（规则+人工） | 固定成本 | >1分钟 | 差 |

### 8.2 成本控制

```markdown
┌─────────────────────────────────────────────────────────────┐
│                    成本控制策略                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 优先使用低成本模型兜底                                  │
│  2. 降级时限制L2/L3操作                                    │
│  3. 及时恢复主服务避免人工成本                              │
│  4. 记录降级日志用于成本分析                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 9. 实施建议

### 9.1 MVP阶段

```
MVP阶段建议：
├── 仅实现Level 1-2（M2.7 → M2.5）
├── 暂不支持跨厂商切换
├── 规则引擎先不做
└── 人工处理仅电话支持

快速上线优先
```

### 9.2 成长阶段

```
成长阶段增加：
├── Level 3支持（豆包/通义）
├── 规则引擎基础版
├── 自动切换逻辑
└── 监控告警
```

### 9.3 成熟阶段

```
成熟阶段完善：
├── 完善的规则引擎
├── 7x24值班
├── SLA保障
└── 成本优化
```
