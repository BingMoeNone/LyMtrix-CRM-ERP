# 整体技术架构设计

> 本文档描述项目的整体技术架构设计，包括七层架构、关键设计决策、技术选型依据。

---

## 1. 架构设计原则

### 1.1 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                    架构设计四大理念                           │
├─────────────────────────────────────────────────────────────┤
│  1. 四层隔离：层层防护，AI无法绕过                            │
│                                                             │
│  2. 单向流转：请求自上而下流动，无反向调用                     │
│                                                             │
│  3. 唯一入口：只有业务引擎可以操作数据库                       │
│                                                             │
│  4. 全链路可控：所有操作可追溯、可复盘                         │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 设计权衡

| 权衡项 | 选择 | 理由 |
|--------|------|------|
| 性能 vs 安全性 | 选择安全性 | 企业级应用，数据安全优先 |
| 复杂度 vs 可维护性 | 接受一定复杂度 | 通过模块化降低维护成本 |
| AI能力 vs 可控性 | 选择可控性 | AI只是辅助，决策权在人 |
| 开发速度 vs 扩展性 | 先保证核心 | MVP阶段聚焦核心功能 |

### 2.2 各层职责矩阵

| 层级 | 名称 | 核心职责 | 绝对不能做什么 |
|------|------|----------|----------------|
| 1 | 多端接入适配层 | 数据格式适配 | 处理业务逻辑 |
| 2 | 统一API网关 | 鉴权、限流、路由 | 修改请求参数 |
| 3 | 多Agent协同决策域 | 意图拆解、多Agent校验 | 直接操作数据库 |
| 4 | OpenClaw调度层 | 工具调用、审批流 | 绕过业务规则 |
| 5 | MCP工具白名单层 | 参数校验、权限管控 | 执行未授权操作 |
| 6 | 业务规则中枢 | 业务逻辑校验、规则执行 | 绕过权限检查 |
| 7 | 引擎核心底座 | 数据访问、权限控制 | 无限制操作 |

## 3. 技术选型

### 3.1 技术栈总览

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| **前端框架** | Vue 3 / React | 生态成熟，社区活跃 |
| **移动端** | uni-app | 一套代码多端运行 |
| **API网关** | Nginx / Spring Cloud Gateway | 成熟稳定 |
| **AI模型** | MiniMax M2.7（主Agent）/ MiniMax M2.5（子Agent） | 强大推理能力，成本优化 |
| **Agent内核** | OpenClaw（内核定制） | MIT协议，仅使用其调度能力，核心逻辑自主开发 |
| **工具协议** | MCP | 标准化，社区支持 |
| **后端框架** | FastAPI / Spring Boot | 性能与开发效率平衡 |
| **ORM** | MyBatis-Plus | 轻量、灵活 |
| **数据库** | MySQL 8.0 | 成熟稳定，场景适配 |
| **缓存** | Redis | 高性能，场景丰富 |
| **部署** | Docker + K8s | 容器化，弹性伸缩 |

### 3.2 数据库选型

| 数据库 | 用途 | 说明 |
|--------|------|------|
| MySQL 8.0 | 业务数据 | 主力数据库 |
| Redis | 缓存、会话、队列 | 高性能场景 |
| MongoDB | 日志存储（可选） | 非结构化数据 |

### 3.3 中间件选型

| 中间件 | 用途 | 说明 |
|--------|------|------|
| RabbitMQ | 消息队列 | 异步任务、解耦 |
| Elasticsearch | 日志搜索 | ELK stack |
| MinIO | 对象存储 | 文件存储 |

## 4. 关键设计决策

### 4.1 为什么选择 OpenClaw（作为内核）

| 考量 | 分析 |
|------|------|
| **协议** | MIT，最宽松的开源协议，适合商业化 |
| **定位** | 作为内核使用，仅依赖其Agent调度、Session管理能力 |
| **核心逻辑** | 投票决策、风险评估、确认工作流、审计日志完全自主开发 |
| **可替换性** | 如OpenClaw不满足需求，可替换为其他方案 |

**重要说明**：本项目以OpenClaw为内核进行深度定制开发，核心业务逻辑（分层投票器、风险评估引擎、白名单管理器、人工确认工作流、业务规则引擎、审计日志系统）完全自主实现，不依赖OpenClaw的业务逻辑能力。

### 4.2 为什么选择 MCP

| 考量 | 分析 |
|------|------|
| **标准化** | 社区认可的标准协议 |
| **隔离性** | AI与业务逻辑完全隔离 |
| **扩展性** | 易于添加新工具 |

### 4.3 为什么不用 LangChain

| 对比项 | OpenClaw | LangChain |
|--------|----------|-----------|
| 定位 | Agent开发框架 | LLM应用库 |
| 多Agent支持 | 原生 | 需要额外集成 |
| Human-in-the-Loop | 原生 | 需要额外开发 |
| 商业友好度 | MIT | Apache 2.0 |

## 8. 扩展性设计

### 8.1 水平扩展

各层均支持水平扩展：
- API网关：增加节点
- Agent服务：增加节点
- 规则引擎：增加节点
- 数据库：分库分表

### 8.2 插件扩展

新增行业或功能时：
1. 开发新的行业插件
2. 注册到插件注册表
3. 无需修改核心代码

---

## 9. 结论

整体技术架构遵循"安全第一"的原则，通过七层隔离实现了：
1. AI能力与业务逻辑的彻底分离
2. 多层防护，无单点失效
3. 全链路可控可追溯
4. 良好的扩展性
