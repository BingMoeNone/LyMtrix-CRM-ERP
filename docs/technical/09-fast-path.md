# 快速通道与白名单机制

> 本文档描述如何通过快速通道和白名单机制，在保证安全性的前提下提升系统响应速度、降低运营成本。

---

## 1. 设计背景

### 1.1 问题

```
传统投票流程的问题：
- 每次操作都要走完整投票
- 即使99%的情况都通过，也要等
- 浪费API调用成本
- 用户等待时间长
```

### 1.2 解决思路

```
快速通道设计：
- 识别"高信任"用户和操作
- 对符合条件的情况跳过投票
- 既保证安全，又提升效率
```

## 2. 白名单机制

### 2.1 白名单定义

```
┌─────────────────────────────────────────────────────────────┐
│                    白名单用户标准                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  进入白名单的条件（满足任意一条）：                           │
│  ├── 历史操作无否决记录                                     │
│  ├── 通过率 > 95%                                         │
│  ├── 月交易额 > 50万                                       │
│  └── 连续3个月无异常操作                                   │
│                                                             │
│  白名单用户享受：                                           │
│  ├── L1操作跳过投票                                       │
│  ├── 专属客服支持                                         │
│  └── 更高额度                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 白名单等级

| 等级 | 名称 | 享受特权 | 进入条件 |
|------|------|----------|----------|
| **WL1** | 普通白名单 | L1跳过投票 | 通过率>95% |
| **WL2** | 高级白名单 | L1+L2各跳过1票 | 通过率>98%+月交易>10万 |
| **WL3** | 核心白名单 | L1+L2全跳过 | 通过率>99%+年交易>100万 |

### 2.3 白名单管理

```markdown
┌─────────────────────────────────────────────────────────────┐
│                    白名单生命周期                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  进入：满足条件 → 运营审核 → 自动加入                        │
│                                                             │
│  维持：每季度评估一次                                       │
│  ├── 条件仍满足 → 继续维持                                 │
│  └── 条件不满足 → 降级或移出                               │
│                                                             │
│  移出：出现否决/异常 → 自动移出 → 需重新申请               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. 快速通道规则

### 3.1 快速通道条件

| 条件 | 说明 | 必须满足 |
|------|------|----------|
| 用户在白名单内 | WL1/WL2/WL3 | 是 |
| 操作类型 | 仅为L1或L2 | 是 |
| 历史通过率 | >该等级阈值 | 是 |
| 金额阈值 | <该等级阈值 | 是 |
| 非异常时段 | 非凌晨/非维护期 | 否 |

### 3.2 快速通道对照表

| 用户等级 | L0 | L1 | L2 |
|----------|----|----|-----|
| **普通用户** | 正常流程 | 正常投票 | 正常投票 |
| **WL1** | 正常 | 跳过投票 | 正常投票 |
| **WL2** | 正常 | 跳过投票 | 跳过1票 |
| **WL3** | 正常 | 跳过投票 | 跳过2票（全过） |

### 3.3 快速通道流程

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────┐
│  检查白名单                                 │
│  └─ 不在白名单 → 正常投票流程               │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│  检查快速通道条件                           │
│  └─ 不满足条件 → 正常投票流程               │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│  执行快速通道                               │
│  → 直接执行或减少投票轮次                  │
└─────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────┐
│  记录快速通道日志                          │
│  → 便于后续审计和分析                      │
└─────────────────────────────────────────────┘
```

## 4. 降级机制

### 4.1 自动降级

```
触发条件：
- 单日否决次数 > 3
- 连续3次操作被否决
- 操作金额异常（>历史平均值5倍）
- 检测到异常操作模式

降级处理：
- 移出白名单
- 加强监控
- 通知用户
```

### 4.2 手动申诉

```
用户可申诉恢复白名单：
1. 提交申诉说明
2. 运营审核
3. 通过后恢复（观察期1个月）
```

## 5. 预判机制

### 5.1 思路

```
用户说："帮我查一下伟业公司的订单"

传统流程：
用户输入 → 风险评估 → Agent投票 → 执行 → 返回
                  ↑
            这里要等

预判流程：
用户输入 → 预判意图 → 预加载权限 → 执行 → 返回
         ↑
    还没等用户确认就开始准备了
```

### 5.2 预判规则

| 关键词 | 预判类型 | 预加载 |
|--------|----------|--------|
| "查..."、"看看..."、"有没有..." | L0查询 | 权限预检 |
| "创建..."、"新增..."、"开一个..." | L1创建 | 数据预验证 |
| "修改..."、"更新..."、"改成..." | L1/L2更新 | 依赖检查 |
| "删除..."、"取消..."、"清空..." | L2删除 | 二次确认 |

### 5.3 预判执行条件

```
预判但不执行的条件：
1. 用户明确发送执行指令
2. 预判结果与用户意图匹配度 > 80%
3. 预判操作已获得足够投票

预判失败的兜底：
- 预判失败 → 回退到正常流程
- 不影响用户体验
```

## 6. 效果预估

### 6.1 性能提升

| 场景 | 正常流程 | 快速通道 | 提升 |
|------|----------|----------|------|
| L0查询 | 1s | 1s | - |
| L1普通 | 2s | 0.5s | 75% |
| L2重要 | 3s | 1.5s | 50% |

### 6.2 成本降低

| 场景 | 正常流程 | 快速通道 | 节省 |
|------|----------|----------|------|
| L1普通 | 2次API | 0次API | 100% |
| L2重要 | 3次API | 1次API | 67% |

### 6.3 用户体验

```
白名单用户感受：
- 90%的操作秒级响应
- 仅高风险操作需要等待
- 专属客服支持
- 更高操作额度
```

## 7. 实施计划

### 7.1 Phase 1（MVP）

```
目标：实现基础白名单
1. 仅支持WL1等级（通过率>95%）
2. 仅L1操作可跳过投票
3. 手动管理白名单列表
```

### 7.2 Phase 2

```
目标：完善白名单机制
1. 支持WL2等级
2. L2操作可跳过1票
3. 自动白名单评估
```

### 7.3 Phase 3

```
目标：引入预判机制
1. 关键词预判
2. 权限预加载
3. 数据预验证
```

## 8. 监控与告警

### 8.1 快速通道监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 快速通道使用率 | 白名单用户中使用快速通道的比例 | <50%（过低） |
| 快速通道成功率 | 快速通道执行后无异常的比例 | <99% |
| 白名单用户否决率 | 白名单用户被否决的比例 | >5% |

### 8.2 异常检测

```markdown
异常模式：
- 同一用户快速通道使用率突然下降
- 白名单用户短时间内频繁被否决
- 预判准确率突然下降
```
