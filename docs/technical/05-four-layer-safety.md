# 六重防护机制详解

> 本文档详细描述项目设计的六重AI安全防护体系，融合 Hook 机制，实现从输入到执行的全链路防护。

---

## 1. 概述

### 1.1 为什么需要六重防护

AI Agent 在企业级应用中面临的核心风险是"幻觉"——AI可能生成错误的理解、参数或决策。在关键业务场景中，这些错误可能导致：
- 错误创建订单
- 错误扣减库存
- 错误转账付款
- 泄露客户数据

单一防护机制无法完全杜绝这些风险，需要多层防御体系。结合 Claude Code 的 Hook 设计，本项目实现了**六重防护 + Hook 体系**。

### 1.2 六重防护体系

```
┌─────────────────────────────────────────────────────────────────┐
│                      六重防护 + Hook 体系                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  【第1重】Pre-Processing Hooks                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  输入清洗 → 频率控制 → 恶意检测（SQL注入、命令注入）      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│  【第2重】多Agent投票校验                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  主Agent + 子Agent1 + 子Agent2 → 分层投票决策           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│  【第3重】Pre-Agent Hooks                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  风险等级确认 → 权限检查 → 操作预检                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│  【第4重】Stop Hooks（核心拦截点）                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  危险模式检测 → 敏感数据脱敏 → 业务规则校验              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│  【第5重】人工确认                                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  用户二次确认 → 执行确认 → 风险提示                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                  │
│  【第6重】Tool/Post-Execution Hooks                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  参数深度验证 → 结果验证 → 审计日志                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 Hook 在防护中的核心作用

| 作用 | 说明 |
|------|------|
| **细粒度控制** | 在 token 级检测危险模式，而非仅操作级 |
| **多时机拦截** | 执行前、执行中、执行后均可干预 |
| **动态调整** | 根据上下文启用/禁用不同 Hook |
| **智能修正** | 拦截后可让 AI 重新生成安全方案 |

> **Hook 系统详细设计见**：[12-hook-system-design.md](12-hook-system-design.md)

## 2. 第一重：Pre-Processing Hooks

### 2.1 设计原理

用户输入是系统安全的**第一道防线**。通过在 AI 处理前对输入进行清洗、验证和过滤，可以拦截大部分恶意请求。

### 2.2 内置 Pre-Processing Hooks

| Hook | 功能 | 拦截类型 |
|------|------|----------|
| `InputSanitizationHook` | 清洗 SQL 注入、命令注入、Prompt 注入 | 100% |
| `RateLimitHook` | 频率控制，防止滥用 | - |

## 3. 第二重：多Agent交叉校验

### 2.1 设计原理

单一AI模型的推理过程可能存在系统性偏差。通过让多个Agent从不同视角独立分析同一问题，可以有效识别和过滤错误理解。

### 2.2 分层投票规则

| 风险等级 | 操作类型 | 通过条件 | 人工确认 | 延迟目标 |
|----------|----------|----------|----------|----------|
| **L0** | 查询类 | 无需投票 | 无 | <1s |
| **L1** | 普通创建/修改（<1万） | 主Agent通过 | 无 | <2s |
| **L2** | 重要/高风险（>1万） | 主+任一子Agent通过（2票） | 强确认 | <3s |

### 2.3 关键机制

1. **独立推理**：三个Agent使用独立的模型实例和上下文
2. **并行校验**：子Agent校验并行执行，避免顺序影响
3. **分层调用**：按需调用，节省成本
4. **一票否决**：任意1个Agent驳回即终止

### 2.4 与 Hook 联动

```
用户输入 → Pre-Processing Hooks → [拦截] → 返回错误
                                → [通过] → 进入多Agent校验
```

### 2.5 实现要点

详见 [01-multi-agent-design.md](01-multi-agent-design.md)

## 4. 第三重：Pre-Agent Hooks + MCP白名单

### 3.1 设计原理

即使AI理解正确，仍可能生成格式错误或超出范围的参数。通过MCP工具定义和参数校验，在AI调用工具前拦截无效请求。

### 3.2 防护层次

```
AI生成方案
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第1层：MCP工具定义校验                                       │
│  → 参数类型、格式、必填项、枚举值                             │
│  → 失败则直接拦截，不进入业务逻辑                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 防护效果

| 错误类型 | 拦截率 |
|----------|--------|
| 类型错误（如字符串传入数字） | 100% |
| 格式错误（如手机号格式） | 100% |
| 范围错误（如超出枚举值） | 100% |
| 必填项缺失 | 100% |

### 3.4 关键机制

1. **白名单管控**：AI只能调用已注册的工具
2. **参数强校验**：每个工具定义严格的参数规范
3. **前置拦截**：校验在调用业务引擎前完成

### 4.4 与 Hook 联动

Pre-Agent Hooks 在多 Agent 投票通过后、Stop Hooks 执行前进行额外检查：

```typescript
// Pre-Agent Hooks 示例
const preAgentHooks = [
  riskLevelConfirmHook,   // 确认风险等级是否正确
  permissionCheckHook,     // 确认用户权限
  operationPreviewHook,    // 生成操作预览
]
```

详见 [03-mcp-tool-whitelist.md](03-mcp-tool-whitelist.md)
详见 [12-hook-system-design.md](12-hook-system-design.md) 第 3.2 节

## 5. 第四重：Stop Hooks + 业务规则校验

### 4.1 设计原理

即使参数格式正确，业务逻辑上可能仍然无效。例如：客户ID格式正确但客户已被禁用，库存数量格式正确但实际库存不足。

### 4.2 防护效果

| 错误类型 | 拦截率 |
|----------|--------|
| 客户不存在或已禁用 | 100% |
| 库存不足 | 100% |
| 价格超出允许范围 | 100% |
| 违反业务约束（如信用额度超限） | 100% |

### 4.3 关键机制

1. **规则固化**：所有业务规则提前定义，不允许AI动态修改
2. **唯一入口**：只有业务引擎可以执行数据操作
3. **全量校验**：所有规则必须100%通过才能执行

### 5.4 Stop Hooks 核心实现

Stop Hooks 是**最终拦截点**，在执行前最后一次检查：

```typescript
// 核心 Stop Hooks
const stopHooks = [
  dangerousPatternHook,      // 危险模式检测（rm -rf, drop table 等）
  sensitiveDataExposureHook, // 敏感数据脱敏
  businessRuleViolationHook,  // 业务规则校验
]

// 执行逻辑
for (const hook of stopHooks) {
  const result = hook.execute(context)
  if (result.status === 'block') {
    // 阻止执行，可选择让 AI 重新生成
    return { blocked: true, message: result.message, allowRetry: true }
  }
}
```

## 6. 第五重：人工确认

### 5.1 设计原理

即使前三层防护全部通过，仍需要人类最终确认。这是"人在回路"原则的体现——AI只有建议权，最终决策权属于用户。

### 5.2 人工确认范围

| 操作类型 | 是否需要确认 | 确认方式 |
|----------|--------------|----------|
| 只读查询 | 否 | 直接返回 |
| 创建操作 | 是 | 二次确认弹窗 |
| 更新操作 | 是 | 变更预览+确认 |
| 删除操作 | 是 | 二次确认+风险提示 |
| 批量操作 | 是 | 逐条确认或批量确认 |

### 5.3 确认流程

```
AI方案通过三层校验
    │
    ▼
生成确认信息
├── 操作类型
├── 关键参数（客户名、金额、数量等）
├── 执行后果预览
└── 风险提示（如有）
    │
    ▼
用户确认/驳回
    │
    ├─ 确认 → 执行操作 → 返回结果
    │
    └─ 驳回 → 终止 → 返回原因
```

### 5.4 全链路审计

所有操作全程留痕：

| 记录内容 | 说明 |
|----------|------|
| 谁 | 用户ID、角色、操作者IP |
| 何时 | 时间戳、时区 |
| 做什么 | 操作类型、实体、ID |
| 什么内容 | 变更前数据、变更后数据 |
| 结果如何 | 成功/失败、错误信息 |

## 7. 第六重：Tool Hooks + Post-Execution Hooks

### 7.1 设计原理

即使人工确认通过，在**工具执行时**和**执行后**仍需验证。这是 Claude Code Hook 体系的核心优势——**运行时保护**。

### 7.2 Tool Hooks

| Hook | 时机 | 功能 |
|------|------|------|
| `ToolParameterValidationHook` | 工具调用前 | 参数深度验证 |
| `ToolExecutionMonitorHook` | 工具执行中 | 监控执行状态 |

### 7.3 Post-Execution Hooks

| Hook | 功能 |
|------|------|
| `ExecutionResultValidationHook` | 验证返回数据合理性 |
| `AuditLogHook` | 记录审计日志 |

## 8. 各层防护协同

### 8.1 防护流程图

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│                    第一重：Pre-Processing Hooks                  │
│  输入清洗 → 频率控制 → 恶意检测                                  │
│  失败：返回错误                                                  │
└─────────────────────────────────────────────────────────────────┘
    │ 通过
    ▼
┌─────────────────────────────────────────────────────────────────┐
│                    第二重：多Agent校验                            │
│  主Agent意图拆解 → 并行分发子Agent → 投票决策                    │
│  失败：返回驳回原因                                              │
└─────────────────────────────────────────────────────────────────┘
    │ 通过
    ▼
┌─────────────────────────────────────────────────────────────────┐
│                    第三重：Pre-Agent Hooks                        │
│  风险等级确认 → 权限检查 → 操作预检                             │
│  失败：返回错误/要求补充                                         │
└─────────────────────────────────────────────────────────────────┘
    │ 通过
    ▼
┌─────────────────────────────────────────────────────────────────┐
│                    第四重：Stop Hooks                            │
│  危险模式检测 → 敏感数据脱敏 → 业务规则校验                      │
│  失败：注入错误 → AI重新生成 或 阻止执行                        │
└─────────────────────────────────────────────────────────────────┘
    │ 通过
    ▼
┌─────────────────────────────────────────────────────────────────┐
│                    第五重：人工确认                              │
│  生成确认信息 → 用户确认/驳回                                   │
│  驳回：返回驳回原因                                              │
└─────────────────────────────────────────────────────────────────┘
    │ 确认
    ▼
┌─────────────────────────────────────────────────────────────────┐
│                    第六重：Tool/Post-Execution Hooks             │
│  参数验证 → 工具执行 → 结果验证 → 审计日志                       │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
返回结果
```

### 8.2 失败处理策略

| 失败层级 | 失败响应 | 可重试 | 用户体验 |
|----------|----------|--------|----------|
| 第一重 | 输入清洗失败 | 否 | "检测到恶意输入" |
| 第二重 | 投票否决 | 否 | "无法完成操作：[驳回原因]" |
| 第三重 | Pre-Agent 检查失败 | 是 | "风险等级不匹配，请确认" |
| 第四重 | Stop Hooks 拦截 | 是 | "危险操作被拦截，可重新生成" |
| 第五重 | 用户驳回 | 否 | "操作已取消" |
| 第六重 | 验证失败 | 是 | "执行结果异常，自动重试" |

### 8.3 Hook 短路机制

```
Stop Hooks 执行时，遇到 Block 结果立即短路：

Hook1 (Block) → 返回，不执行后续 Hooks
Hook2 (Allow) → 不执行
Hook3 (Allow) → 不执行
```

## 9. 防护效能评估

### 9.1 整体拦截效果

假设各层独立错误率：

| 防护层 | 错误通过率 |
|--------|------------|
| 第一重（Pre-Processing Hooks） | 30% |
| 第二重（多Agent） | 1% |
| 第三重（Pre-Agent Hooks） | 5% |
| 第四重（Stop Hooks） | 0% |
| 第五重（人工确认） | 0%（用户可识别剩余错误） |
| 第六重（Tool/Post-Execution Hooks） | 0% |

**综合错误率**：30% × 1% × 5% × 0% × 0% × 0% ≈ 0%

实际上 Stop Hooks 可拦截 99% 的剩余风险，Hook 的短路机制确保危险操作不会漏过。

### 9.2 Hook 特有优势

| 指标 | 纯提示词方案 | 本项目方案（含 Hook） |
|------|-------------|---------------------|
| 危险模式检测 | 依赖模型理解，可能忽略 | 正则/规则匹配，100% 检测 |
| 拦截时机 | 执行后 | **执行前** |
| 敏感数据泄露 | 可能泄露 | 自动脱敏 |
| 频率攻击 | 无防护 | 限流保护 |
| AI 自我修正 | 依赖模型 | 强制重新生成 |

### 9.3 场景模拟

**场景：用户说"给某客户加100万信用额度"**

| 防护层 | 检验内容 | 拦截结果 |
|--------|----------|----------|
| 第一重 | Pre-Processing 通过 | 进入下一层 |
| 第二重 | 子Agent1发现额度超出规则限制 | 驳回：超额 |
| 第二重 | 子Agent2发现用户无此权限 | 驳回：无权限 |
| → | 投票否决 | 不进入后续 |

**场景：AI误以为客户可用余额充足**

| 防护层 | 检验内容 | 拦截结果 |
|--------|----------|----------|
| 第一重 | Pre-Processing 通过 | 进入下一层 |
| 第二重 | 多Agent校验通过 | 进入下一层 |
| 第三重 | Pre-Agent 通过 | 进入下一层 |
| 第四重 | Stop Hooks 通过 | 进入下一层 |
| 第五重 | 人工确认通过 | 进入下一层 |
| 第六重 | Tool 执行成功 | 返回结果 |

**场景：用户输入包含 SQL 注入**

| 防护层 | 检验内容 | 拦截结果 |
|--------|----------|----------|
| 第一重 | `InputSanitizationHook` 检测到 SQL 注入 | **立即拦截** |
| → | 返回错误，不进入后续 | 阻止成功 |

## 10. 性能影响

### 10.1 各层延迟

| 防护层 | 典型延迟 | 说明 |
|--------|----------|------|
| 第一重 | < 1ms | 本地正则匹配，无网络 |
| 第二重 | 2-5秒 | 3个模型并行调用 |
| 第三重 | < 5ms | 本地 Hook 执行 |
| 第四重 | < 10ms | Stop Hooks 短路执行 |
| 第五重 | 用户交互时间 | 人工确认（不计时） |
| 第六重 | < 100ms | 工具执行 + 结果验证 |

### 10.2 优化策略

| 优化点 | 策略 |
|--------|------|
| 多Agent并行 | 3个Agent同时调用，减少等待 |
| 参数校验缓存 | 热点规则结果缓存 |
| 业务规则异步 | 非关键规则异步执行 |
| 确认流程优化 | 简化确认UI，减少用户思考时间 |

---

## 11. 结论

六重防护体系通过**层层拦截 + Hook 机制**，可以从根本上杜绝AI幻觉对企业级业务的影响：

1. **第一重**：Pre-Processing Hooks，输入清洗防注入
2. **第二重**：多Agent从不同视角独立校验，过滤99%问题
3. **第三重**：Pre-Agent Hooks，风险等级确认
4. **第四重**：Stop Hooks，危险模式实时检测（核心拦截点）
5. **第五重**：人工确认作为最终安全网
6. **第六重**：Tool/Post-Execution Hooks，执行时保护

即使极端情况下AI仍然出错，Stop Hooks 和人工确认双重保障确保不会造成实际业务损失。
