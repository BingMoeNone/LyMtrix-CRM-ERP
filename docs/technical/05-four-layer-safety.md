# 四重防护机制详解

> 本文档详细描述项目设计的四重AI幻觉防护体系，从事前到事后，层层拦截，确保AI操作的安全可控。

---

## 1. 概述

### 1.1 为什么需要四重防护

AI Agent 在企业级应用中面临的核心风险是"幻觉"——AI可能生成错误的理解、参数或决策。在关键业务场景中，这些错误可能导致：
- 错误创建订单
- 错误扣减库存
- 错误转账付款
- 泄露客户数据

单一防护机制无法完全杜绝这些风险，需要多层防御体系。

### 1.2 四重防护体系

```
┌─────────────────────────────────────────────────────────────┐
│                     四重防护体系                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  事前防护                                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │          多Agent交叉校验（过滤99%幻觉）              │   │
│  └─────────────────────────────────────────────────────┘   │
│                            │                               │
│                            ▼                               │
│  事中防护1                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │          MCP白名单+参数校验                          │   │
│  └─────────────────────────────────────────────────────┘   │
│                            │                               │
│                            ▼                               │
│  事中防护2                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │          业务规则最终校验                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                            │                               │
│                            ▼                               │
│  事后防护                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │          人工确认+全链路审计                         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. 第一重：事前防护 - 多Agent交叉校验

### 2.1 设计原理

单一AI模型的推理过程可能存在系统性偏差。通过让多个Agent从不同视角独立分析同一问题，可以有效识别和过滤错误理解。

### 2.2 防护效果

| 幻觉类型 | 单Agent错误率 | 多Agent校验后错误率 |
|----------|---------------|---------------------|
| 意图理解错误 | 5-10% | < 1% |
| 参数生成错误 | 3-8% | < 1% |
| 违规操作提议 | 2-5% | < 0.5% |

### 2.3 关键机制

1. **独立推理**：三个Agent使用独立的模型实例和上下文
2. **并行校验**：子Agent校验并行执行，避免顺序影响
3. **全部通过**：3票全部通过才执行
4. **一票否决**：任意1个Agent驳回即终止

详见 [01-multi-agent-design.md](01-multi-agent-design.md)

## 3. 第二重：事中防护1 - MCP白名单+参数校验

### 3.1 设计原理

即使AI理解正确，仍可能生成格式错误或超出范围的参数。通过MCP工具定义和参数校验，在AI调用工具前拦截无效请求。

### 3.2 防护层次

```
AI生成方案
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  第1层：MCP工具定义校验                                       │
│  → 参数类型、格式、必填项、枚举值                             │
│  → 失败则直接拦截，不进入业务逻辑                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 防护效果

| 错误类型 | 拦截率 |
|----------|--------|
| 类型错误（如字符串传入数字） | 100% |
| 格式错误（如手机号格式） | 100% |
| 范围错误（如超出枚举值） | 100% |
| 必填项缺失 | 100% |

### 3.4 关键机制

1. **白名单管控**：AI只能调用已注册的工具
2. **参数强校验**：每个工具定义严格的参数规范
3. **前置拦截**：校验在调用业务引擎前完成

### 3.5 实现要点

详见 [03-mcp-tool-whitelist.md](03-mcp-tool-whitelist.md)

## 4. 第三重：事中防护2 - 业务规则最终校验

### 4.1 设计原理

即使参数格式正确，业务逻辑上可能仍然无效。例如：客户ID格式正确但客户已被禁用，库存数量格式正确但实际库存不足。

### 4.2 防护效果

| 错误类型 | 拦截率 |
|----------|--------|
| 客户不存在或已禁用 | 100% |
| 库存不足 | 100% |
| 价格超出允许范围 | 100% |
| 违反业务约束（如信用额度超限） | 100% |

### 4.3 关键机制

1. **规则固化**：所有业务规则提前定义，不允许AI动态修改
2. **唯一入口**：只有业务引擎可以执行数据操作
3. **全量校验**：所有规则必须100%通过才能执行

### 4.4 实现要点

详见 [04-business-engine.md](04-business-engine.md)

## 5. 第四重：事后防护 - 人工确认+全链路审计

### 5.1 设计原理

即使前三层防护全部通过，仍需要人类最终确认。这是"人在回路"原则的体现——AI只有建议权，最终决策权属于用户。

### 5.2 人工确认范围

| 操作类型 | 是否需要确认 | 确认方式 |
|----------|--------------|----------|
| 只读查询 | 否 | 直接返回 |
| 创建操作 | 是 | 二次确认弹窗 |
| 更新操作 | 是 | 变更预览+确认 |
| 删除操作 | 是 | 二次确认+风险提示 |
| 批量操作 | 是 | 逐条确认或批量确认 |

### 5.3 确认流程

```
AI方案通过三层校验
    │
    ▼
生成确认信息
├── 操作类型
├── 关键参数（客户名、金额、数量等）
├── 执行后果预览
└── 风险提示（如有）
    │
    ▼
用户确认/驳回
    │
    ├─ 确认 → 执行操作 → 返回结果
    │
    └─ 驳回 → 终止 → 返回原因
```

### 5.4 全链路审计

所有操作全程留痕：

| 记录内容 | 说明 |
|----------|------|
| 谁 | 用户ID、角色、操作者IP |
| 何时 | 时间戳、时区 |
| 做什么 | 操作类型、实体、ID |
| 什么内容 | 变更前数据、变更后数据 |
| 结果如何 | 成功/失败、错误信息 |

### 5.5 审计日志不可篡改

```typescript
// 审计日志写入策略
class AuditLogWriter {
  async write(entry: AuditLog): Promise<void> {
    // 1. 写入审计数据库（append-only）
    await this.auditDb.insert({
      ...entry,
      // 2. 附加防篡改字段
      hash: this.calculateHash(entry),
      sequence: await this.getNextSequence(),
    });

    // 3. 定期同步到冷存储
    await this.archiveOldLogs();
  }
}
```

## 6. 各层防护协同

### 6.1 防护流程图

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│                    第一重：多Agent校验                         │
│  主Agent意图拆解 → 并行分发子Agent → 投票决策                  │
│  失败：返回驳回原因                                            │
└─────────────────────────────────────────────────────────────┘
    │ 通过
    ▼
┌─────────────────────────────────────────────────────────────┐
│                    第二重：MCP白名单校验                       │
│  检查工具是否在白名单 → 参数类型/格式/范围校验                  │
│  失败：返回参数错误详情                                        │
└─────────────────────────────────────────────────────────────┘
    │ 通过
    ▼
┌─────────────────────────────────────────────────────────────┐
│                    第三重：业务规则校验                        │
│  业务引擎执行规则校验 → 业务逻辑有效性检查                      │
│  失败：返回违反的规则                                          │
└─────────────────────────────────────────────────────────────┘
    │ 通过（写入操作）
    ▼
┌─────────────────────────────────────────────────────────────┐
│                    第四重：人工确认                           │
│  生成确认信息 → 用户确认/驳回                                │
│  驳回：返回驳回原因                                           │
└─────────────────────────────────────────────────────────────┘
    │ 确认
    ▼
执行操作 → 审计日志 → 返回结果
```

### 6.2 失败处理策略

| 失败层级 | 失败响应 | 用户体验 |
|----------|----------|----------|
| 第一重 | 投票否决 | "无法完成操作：[驳回原因]" |
| 第二重 | 参数校验失败 | "输入有误：[具体字段错误]" |
| 第三重 | 业务规则不通过 | "业务规则校验失败：[规则名称]" |
| 第四重 | 用户驳回 | "操作已取消" |

## 7. 防护效能评估

### 7.1 整体拦截效果

假设各层独立错误率：

| 防护层 | 错误通过率 |
|--------|------------|
| 第一重（多Agent） | 1% |
| 第二重（MCP校验） | 0% |
| 第三重（业务规则） | 0% |
| 第四重（人工确认） | 0%（用户可识别剩余错误） |

**综合错误率**：1% × 0% × 0% × 0% ≈ 0%

实际上人工确认层可以拦截剩余的1%，综合错误率接近0%。

### 7.2 场景模拟

**场景：用户说"给某客户加100万信用额度"**

| 防护层 | 检验内容 | 拦截结果 |
|--------|----------|----------|
| 第一重 | 子Agent1发现额度超出规则限制 | 驳回：超额 |
| 第一重 | 子Agent2发现用户无此权限 | 驳回：无权限 |
| → | 投票否决 | 不进入后续 |

**场景：AI误以为客户可用余额充足**

| 防护层 | 检验内容 | 拦截结果 |
|--------|----------|----------|
| 第一重 | 多Agent校验通过 | 进入下一层 |
| 第二重 | 参数校验通过 | 进入下一层 |
| 第三重 | 业务规则检查余额 | 驳回：余额不足 |
| → | 终止执行 | 返回错误 |

## 8. 性能影响

### 8.1 各层延迟

| 防护层 | 典型延迟 | 说明 |
|--------|----------|------|
| 第一重 | 2-5秒 | 3个模型并行调用 |
| 第二重 | < 10ms | 本地参数校验 |
| 第三重 | 50-200ms | 数据库校验 |
| 第四重 | 用户交互时间 | 人工确认（不计时） |

### 8.2 优化策略

| 优化点 | 策略 |
|--------|------|
| 多Agent并行 | 3个Agent同时调用，减少等待 |
| 参数校验缓存 | 热点规则结果缓存 |
| 业务规则异步 | 非关键规则异步执行 |
| 确认流程优化 | 简化确认UI，减少用户思考时间 |

---

## 9. 结论

四重防护体系通过层层拦截，可以将从根本上杜绝AI幻觉对企业级业务的影响：

1. **事前防护**：多Agent从不同视角独立校验，过滤99%问题
2. **事中防护1**：MCP白名单+参数校验，拦截格式错误
3. **事中防护2**：业务规则最终校验，拦截逻辑错误
4. **事后防护**：人工确认作为最终安全网

即使极端情况下AI仍然出错，人工确认环节也会拦截，确保不会造成实际业务损失。
