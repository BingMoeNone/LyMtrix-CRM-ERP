# Multi-Agent 协同决策系统设计

> 本文档详细描述 1主2从 多Agent架构的设计方案，包括Agent角色定义、投票决策规则、通信协议、权限边界。

---

## 1. 背景与问题

### 1.1 为什么需要 Multi-Agent

单一 Agent 在复杂业务场景下面临以下问题：

| 问题 | 描述 |
|------|------|
| **意图理解偏差** | 用户自然语言表达模糊，单一Agent容易理解错误 |
| **参数错误** | AI可能生成不符合业务要求的参数 |
| **群体幻觉** | 同一AI的多个推理步骤可能同时出错 |
| **权限失控** | 单一Agent可能执行超出权限的操作 |

### 1.2 解决思路

通过"1主2从"架构，让多个Agent从不同视角独立校验同一方案，通过投票机制达成共识，从根源过滤AI幻觉。

---

## 2. Agent 角色设计

### 2.1 三角色职责矩阵

| Agent | 角色定位 | 核心职责 | 不可突破的权限边界 |
|-------|----------|----------|-------------------|
| **主Agent** | 意图匹配&总调度 | 1. 拆解自然语言意图<br>2. 生成基础执行方案<br>3. 方案分发与投票仲裁<br>4. 调用MCP工具 | 1. 只能调用白名单内的MCP工具<br>2. 绝对不能绕过子Agent校验直接执行<br>3. 无直接写入数据的权限 |
| **子Agent1** | 业务规则合规 | 1. 校验客户是否存在<br>2. 校验库存是否充足<br>3. 校验定价是否合规<br>4. 输出合规方案或驳回 | 1. 只能基于已固化的业务规则校验<br>2. 不能自定义业务规则<br>3. 对不合规方案有一票否决权 |
| **子Agent2** | 安全权限合规 | 1. 校验用户操作权限<br>2. 校验是否需要人工确认<br>3. 校验数据合规要求<br>4. 输出安全方案或驳回 | 1. 只能基于已固化的权限规则校验<br>2. 不能修改权限边界<br>3. 对违规操作有一票否决权 |

### 2.2 Agent 独立性保证

为避免"群体幻觉"，三个Agent必须满足：

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 独立性要求                           │
├─────────────────────────────────────────────────────────────┤
│ 1. 独立模型实例：每个Agent使用独立的模型调用，无共享状态      │
│ 2. 独立上下文：每个Agent仅能看到自己绑定的规则集              │
│ 3. 独立推理：Agent之间不能相互参考推理过程                    │
│ 4. 并行执行：子Agent校验必须并行进行，避免顺序影响            │
└─────────────────────────────────────────────────────────────┘
```

## 3. 分层投票决策规则

### 3.1 分层设计背景

```
为什么不用3票固定制：
- 3票制过于严格，导致大量操作无法执行
- 所有操作都用3票，成本高、延迟大
- 90%的操作是低风险的查询/普通创建

解决方案：分层投票机制
- L0查询：无需投票
- L1普通：主Agent通过即可
- L2重要：主+任一子Agent通过（2票）
```

### 3.2 分层投票规则

| 风险等级 | 操作类型 | 通过条件 | 人工确认 | 说明 |
|----------|----------|----------|----------|------|
| **L0** | 查询类 | 无需投票 | 无 | 主Agent直接返回 |
| **L1** | 普通创建/修改（<1万） | 主Agent通过 | 无 | 自动执行 |
| **L2** | 重要/高风险（>1万） | 主+任一子Agent通过 | 强确认 | 2票通过 |

### 3.3 分层决策流程图

```
用户输入需求
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  风险评估：判断操作属于L0/L1/L2                            │
└─────────────────────────────────────────────────────────────┘
    │
    ├──→ L0 ──────────────────────────────────────→ 主Agent直接返回
    │
    ├──→ L1 ──────────────────────────────────────→ 主Agent投票 → 通过 → 执行
    │
    └──→ L2 ───────────────────┬──────────────────┐
                                ▼                  ▼
                         ┌─────────┐      ┌─────────┐
                         │ 主Agent │      │子Agent1 │
                         │  方案A  │      │  方案B  │
                         └────┬────┘      └────┬────┘
                              │                │
                              └────────┬───────┘
                                       ▼
                               ┌───────────────┐
                               │   2票决策    │
                               └───────────────┘
                                       │
                              ┌─────────┴─────────┐
                              ▼                   ▼
                        ┌─────────┐         ┌─────────┐
                        │  通过   │         │  否决   │
                        │ 2票一致 │         │ 任意1驳回│
                        └────┬────┘         └────┬────┘
                             │                   │
                             ▼                   ▼
                        执行方案            终止执行
                        +强确认
```

### 3.4 三种决策结果

| 决策结果 | 触发条件 | 后续动作 |
|----------|----------|----------|
| **通过** | 对应等级的投票通过 | 进入确认流程 |
| **否决** | 任意1个Agent提出风险驳回 | 终止执行，返回驳回原因和风险点给用户 |
| **分歧** | 方案不一致 | 最多重新分发校验2轮，2轮后仍分歧则交用户手动决策 |

### 3.5 L2的2票制 vs 3票制

| 对比项 | 2票制（主+任一子） | 3票制（全部） |
|--------|---------------------|---------------|
| 安全性 | 中等（仍需2票一致） | 最高 |
| 执行效率 | 高（快速通过） | 低 |
| 分歧率 | 低 | 高 |
| 成本 | 低（M2.5参与） | 高（3次M2.7） |

**结论**：L2使用2票制足够安全，且能大幅提升效率和降低成本。

## 4. 通信协议设计

### 4.1 Agent 间消息格式

```json
{
  "message_id": "uuid-v4",
  "timestamp": "2026-04-16T10:00:00Z",
  "sender": "主Agent|子Agent1|子Agent2",
  "receiver": "主Agent|子Agent1|子Agent2",
  "type": "方案分发|校验结果|投票请求|人工确认触发",
  "payload": {
    "方案ID": "uuid-v4",
    "操作类型": "CREATE_ORDER|QUERY_INVENTORY|...",
    "参数": {},
    "校验结果": "通过|驳回|有条件通过",
    "驳回原因": "string|null",
    "风险点": ["string"]|null
  }
}
```

### 4.2 消息类型定义

| 消息类型 | 发送方 | 接收方 | 说明 |
|----------|--------|--------|------|
| **方案分发** | 主Agent | 子Agent1, 子Agent2 | 主Agent将方案分发给子Agent进行校验 |
| **校验结果** | 子Agent1, 子Agent2 | 主Agent | 子Agent返回校验结果 |
| **投票请求** | 主Agent | 主Agent | 主Agent内部执行投票决策 |
| **人工确认触发** | 主Agent | 前端/用户 | 投票通过后触发用户确认 |

### 4.3 超时与重试机制

| 场景 | 超时时间 | 重试策略 |
|------|----------|----------|
| 子Agent校验超时 | 30秒 | 自动重试1次，仍超时则视为"无法校验" |
| 方案分发失败 | 10秒 | 自动重试3次，仍失败则终止流程 |
| 人工确认超时 | 可配置（默认24小时） | 发送提醒，仍未确认则方案失效 |

┌─────────────────────────────────────────────────────────────┐

│  │  │ 主Agent │  │子Agent1 │  │子Agent2 │             │    │

│  │       └────────────┴────────────┘                   │    │

### 5.2 OpenClaw 插件开发要点

1. **自定义 Skill**：开发 `BusinessRuleValidatorSkill` 和 `SecurityValidatorSkill`

3. **生命周期管理**：利用 OpenClaw 的 Agent 管理能力
4. **日志审计**：通过 OpenClaw 钩子接入审计系统

## 6. 技术实现考量

### 6.1 模型选型

| Agent | 推荐模型 | 理由 | 调用频率 |
|-------|----------|------|----------|
| 主Agent | MiniMax M2.7 | 需要强意图理解和高上下文窗口 | L0/L1/L2都调用 |
| 子Agent1 | MiniMax M2.5 | 只需校验固定规则，不需要强推理 | L1/L2调用 |
| 子Agent2 | MiniMax M2.5 | 只需检查权限配置 | 仅L2调用 |

**成本优化说明**：
- L0仅调用主Agent（节省100%子Agent调用）
- L1调用主Agent+子Agent1（节省50%）
- L2调用全部3个Agent（完整调用保证安全）

### 6.2 Temperature 设置

所有Agent统一设置 `temperature=0.1`，降低AI创造性幻觉，提升输出稳定性。

### 6.3 并行 vs 串行

| 方案 | 优点 | 缺点 |
|------|------|------|
| **并行分发**（推荐） | 延迟低，避免顺序影响 | 需要并发处理能力 |
| 串行分发 | 实现简单 | 延迟高，后面的Agent可能受前面影响 |

---

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 子Agent校验超时 | 中 | 高 | 设置超时重试机制，超时视为"无法校验" |
| 分歧过于频繁 | 中 | 中 | 优化prompt工程，持续迭代规则 |
| 并行调用成本过高 | 高 | 低 | 分层投票+按需调用 |

---

## 8. 结论

Multi-Agent 协同决策系统是本项目防幻觉的核心机制。通过：

1. **分层投票**：L0无投票→L1主1票→L2主+1票，按需调用节省成本
2. **按需模型**：主Agent用M2.7，子Agent用M2.5，成本降低50%
3. **独立推理**：避免群体幻觉
4. **人工确认**：最终安全网

这套机制可以在保证安全性的同时，大幅降低成本和延迟。

## 10. 与 Hook 系统联动

### 10.1 联动架构

Multi-Agent 投票结果与 Hook 系统形成**双重校验**：

```
用户输入
    │
    ▼
Pre-Processing Hooks（输入清洗）
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Multi-Agent 投票                                         │
│  主Agent → 方案生成 → 分发给 Agent1/Agent2              │
│  投票结果：通过/否决/分歧                                 │
└─────────────────────────────────────────────────────────────┘
    │ 通过
    ▼
Pre-Agent Hooks（风险等级确认）
    │
    ▼
Stop Hooks（最终拦截）
    │
    ▼
人工确认 → 执行
```

### 10.2 职责边界

| 阶段 | 负责方 | 说明 |
|------|--------|------|
| 意图理解正确性 | Multi-Agent | 确保 AI 理解了用户要什么 |
| 危险操作检测 | Hooks | 确保操作不危险 |
| 业务合规性 | Multi-Agent + Hooks | 双重确认 |
| 人工授权 | 用户 | 最终确认 |

### 10.3 分工示例

**场景：用户说"删除所有客户"**

| 阶段 | 处理 | 结果 |
|------|------|------|
| Multi-Agent | Agent2 发现权限不足 | **否决**（无需 Hook） |
| Stop Hooks | `dangerousPatternHook` 检测到批量删除 | **拦截** |

**场景：用户说"修改某客户价格"**

| 阶段 | 处理 | 结果 |
|------|------|------|
| Multi-Agent | 投票通过 | 进入下一层 |
| Pre-Agent Hooks | 确认风险等级为 L2 | 强制确认 |
| Stop Hooks | `businessRuleViolationHook` 检查通过 | 通过 |
| 人工确认 | 用户确认 | 执行 |
