# Multi-Agent 协同决策系统设计

> 本文档详细描述 1主2从 多Agent架构的设计方案，包括Agent角色定义、投票决策规则、通信协议、权限边界。

---

## 1. 背景与问题

### 1.1 为什么需要 Multi-Agent

单一 Agent 在复杂业务场景下面临以下问题：

| 问题 | 描述 |
|------|------|
| **意图理解偏差** | 用户自然语言表达模糊，单一Agent容易理解错误 |
| **参数错误** | AI可能生成不符合业务要求的参数 |
| **群体幻觉** | 同一AI的多个推理步骤可能同时出错 |
| **权限失控** | 单一Agent可能执行超出权限的操作 |

### 1.2 解决思路

通过"1主2从"架构，让多个Agent从不同视角独立校验同一方案，通过投票机制达成共识，从根源过滤AI幻觉。

---

## 2. Agent 角色设计

### 2.1 三角色职责矩阵

| Agent | 角色定位 | 核心职责 | 不可突破的权限边界 |
|-------|----------|----------|-------------------|
| **主Agent** | 意图匹配&总调度 | 1. 拆解自然语言意图<br>2. 生成基础执行方案<br>3. 方案分发与投票仲裁<br>4. 调用MCP工具 | 1. 只能调用白名单内的MCP工具<br>2. 绝对不能绕过子Agent校验直接执行<br>3. 无直接写入数据的权限 |
| **子Agent1** | 业务规则合规 | 1. 校验客户是否存在<br>2. 校验库存是否充足<br>3. 校验定价是否合规<br>4. 输出合规方案或驳回 | 1. 只能基于已固化的业务规则校验<br>2. 不能自定义业务规则<br>3. 对不合规方案有一票否决权 |
| **子Agent2** | 安全权限合规 | 1. 校验用户操作权限<br>2. 校验是否需要人工确认<br>3. 校验数据合规要求<br>4. 输出安全方案或驳回 | 1. 只能基于已固化的权限规则校验<br>2. 不能修改权限边界<br>3. 对违规操作有一票否决权 |

### 2.2 Agent 独立性保证

为避免"群体幻觉"，三个Agent必须满足：

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 独立性要求                           │
├─────────────────────────────────────────────────────────────┤
│ 1. 独立模型实例：每个Agent使用独立的模型调用，无共享状态      │
│ 2. 独立上下文：每个Agent仅能看到自己绑定的规则集              │
│ 3. 独立推理：Agent之间不能相互参考推理过程                    │
│ 4. 并行执行：子Agent校验必须并行进行，避免顺序影响            │
└─────────────────────────────────────────────────────────────┘
```

## 3. 投票决策规则

### 3.1 决策流程图

```
用户输入需求
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│                    主Agent：意图拆解                          │
│  拆解自然语言 → 匹配MCP工具 → 生成基础执行方案                │
└─────────────────────────────────────────────────────────────┘
    │
    ├──────────────────┬──────────────────┐
    ▼                  ▼                  ▼
┌─────────┐      ┌─────────┐      ┌─────────┐
│ 主Agent │      │ 子Agent1│      │ 子Agent2│
│  方案A  │      │  方案B  │      │  方案C  │
└────┬────┘      └────┬────┘      └────┬────┘
     │                │                │
     └────────────────┴────────────────┘
                    ▼
            ┌───────────────┐
            │   投票决策    │
            └───────────────┘
                    │
         ┌──────────┼──────────┐
         ▼          ▼          ▼
    ┌─────────┐ ┌─────────┐ ┌─────────┐
    │  通过   │ │  否决   │ │  分歧   │
    │ 3票全部通过 │ │ 任意1驳回 │ │ 3方不一致 │
    └────┬────┘ └────┬────┘ └────┬────┘
         │           │           │
         ▼           ▼           ▼
      执行方案     终止执行    重新校验
                                  │(最多2轮)
                                  ▼
                            仍分歧→交用户决策
```

### 3.2 三种决策结果

| 决策结果 | 触发条件 | 后续动作 |
|----------|----------|----------|
| **通过** | 3个Agent全部通过，核心操作、参数、工具调用完全一致，且无任何驳回 | 进入人工确认流程 |
| **否决** | 任意1个Agent提出风险驳回 | 终止执行，返回驳回原因和风险点给用户 |
| **分歧** | 3个Agent的方案均不一致 | 最多重新分发校验2轮，2轮后仍分歧则交用户手动决策 |

### 3.3 方案一致性判断标准

两个方案被视为"一致"必须满足：

1. **核心操作一致**：最终执行的操作类型相同（如都是"创建订单"）
2. **参数一致**：关键参数值相同或兼容（如客户ID、金额）
3. **工具调用一致**：调用的MCP工具相同
4. **无驳回**：两个方案都没有包含"驳回"标记

## 4. 通信协议设计

### 4.1 Agent 间消息格式

```json
{
  "message_id": "uuid-v4",
  "timestamp": "2026-04-16T10:00:00Z",
  "sender": "主Agent|子Agent1|子Agent2",
  "receiver": "主Agent|子Agent1|子Agent2",
  "type": "方案分发|校验结果|投票请求|人工确认触发",
  "payload": {
    "方案ID": "uuid-v4",
    "操作类型": "CREATE_ORDER|QUERY_INVENTORY|...",
    "参数": {},
    "校验结果": "通过|驳回|有条件通过",
    "驳回原因": "string|null",
    "风险点": ["string"]|null
  }
}
```

### 4.2 消息类型定义

| 消息类型 | 发送方 | 接收方 | 说明 |
|----------|--------|--------|------|
| **方案分发** | 主Agent | 子Agent1, 子Agent2 | 主Agent将方案分发给子Agent进行校验 |
| **校验结果** | 子Agent1, 子Agent2 | 主Agent | 子Agent返回校验结果 |
| **投票请求** | 主Agent | 主Agent | 主Agent内部执行投票决策 |
| **人工确认触发** | 主Agent | 前端/用户 | 投票通过后触发用户确认 |

### 4.3 超时与重试机制

| 场景 | 超时时间 | 重试策略 |
|------|----------|----------|
| 子Agent校验超时 | 30秒 | 自动重试1次，仍超时则视为"无法校验" |
| 方案分发失败 | 10秒 | 自动重试3次，仍失败则终止流程 |
| 人工确认超时 | 可配置（默认24小时） | 发送提醒，仍未确认则方案失效 |

## 5. 与 OpenClaw 的集成

### 5.1 集成架构

```
┌─────────────────────────────────────────────────────────────┐
│                    OpenClaw 调度层                          │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              Multi-Agent 协同决策域                  │    │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐             │    │
│  │  │ 主Agent │  │子Agent1 │  │子Agent2 │             │    │
│  │  └────┬────┘  └────┬────┘  └────┬────┘             │    │
│  │       └────────────┴────────────┘                   │    │
│  │                    │                                 │    │
│  │              投票仲裁器                               │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 OpenClaw 插件开发要点

1. **自定义 Skill**：开发 `BusinessRuleValidatorSkill` 和 `SecurityValidatorSkill`
2. **工具注册**：将子Agent的校验逻辑封装为 OpenClaw 工具
3. **生命周期管理**：利用 OpenClaw 的 Agent 管理能力
4. **日志审计**：通过 OpenClaw 钩子接入审计系统

## 6. 技术实现考量

### 6.1 模型选型

| Agent | 推荐模型 | 理由 |
|-------|----------|------|
| 主Agent | 豆包Pro-32k / 通义千问4 | 需要强意图理解和高上下文窗口 |
| 子Agent1 | 同主Agent | 保证校验质量一致 |
| 子Agent2 | 同主Agent | 保证权限判断准确 |

### 6.2 Temperature 设置

所有Agent统一设置 `temperature=0.1`，降低AI创造性幻觉，提升输出稳定性。

### 6.3 并行 vs 串行

| 方案 | 优点 | 缺点 |
|------|------|------|
| **并行分发**（推荐） | 延迟低，避免顺序影响 | 需要并发处理能力 |
| 串行分发 | 实现简单 | 延迟高，后面的Agent可能受前面影响 |

---

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 子Agent校验超时 | 中 | 高 | 设置超时重试机制，超时视为"无法校验" |
| 三个Agent全部误判 | 低 | 极高 | 人工确认作为最后防线 |
| 投票分歧过于频繁 | 中 | 中 | 优化prompt工程，持续迭代规则 |
| 并行调用成本过高 | 高 | 低 | 考虑串行+缓存优化 |

---

## 8. 结论

Multi-Agent 协同决策系统是本项目防幻觉的核心机制。通过：

1. **角色分离**：主Agent负责调度，子Agent负责校验，各司其职
2. **独立推理**：避免群体幻觉
3. **投票机制**：3票全部通过才执行，任意1驳回直接终止
4. **人工确认**：最终安全网

这套机制可以从源头过滤99%的AI幻觉风险，同时保持系统的高可用性。
