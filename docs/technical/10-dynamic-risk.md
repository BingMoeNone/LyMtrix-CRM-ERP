# 动态风险评估体系

> 本文档描述如何通过动态因子评分，实现风险等级的智能升降级，在保证安全的同时最大化运营效率。

---

## 1. 设计背景

### 1.1 静态分级的问题

```
传统静态分级的问题：
- 金额>10万 → L3
- 不管客户信用、历史表现，一刀切
- 高信用老客户也要走完整流程
- 低信用新客户反而门槛一样
```

### 1.2 动态评估的优势

```
动态风险评估的优势：
- 考虑客户信用、交易历史
- 考虑操作时间、频率
- 考虑实时风险信号
- 实现真正的"风险定价"
```

## 2. 动态因子体系

### 2.1 因子分类

```
┌─────────────────────────────────────────────────────────────┐
│                    动态风险因子                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【基础因子】                                                │
│  ├── 操作类型（query/create/update/delete）                │
│  ├── 金额大小                                             │
│  └── 客户类型（普通/VIP/高风险）                          │
│                                                             │
│  【加分因子】风险上升                                       │
│  ├── 新客户（首次交易）                                    │
│  ├── 非工作时间操作                                        │
│  ├── 短时间频繁操作                                        │
│  ├── 高风险行业客户                                        │
│  └── 异常操作模式                                          │
│                                                             │
│  【减分因子】风险下降                                       │
│  ├── 回头客（多次交易）                                    │
│  ├── 高信用客户                                            │
│  ├── 定期自动操作                                          │
│  └── 白名单客户                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 因子权重表

| 因子 | 方向 | 权重 | 说明 |
|------|------|------|------|
| 操作类型 | - | 高 | delete类权重最高 |
| 金额大小 | - | 高 | 金额越大权重越高 |
| 客户类型 | - | 中 | 高风险客户加权 |
| 新客户 | +1 | 中 | 首次交易不确定性高 |
| 非工作时间 | +1 | 低 | 风险信号 |
| 频繁操作 | +1~2 | 中 | 可能的异常行为 |
| 高风险行业 | +1 | 中 | 行业特性 |
| 异常模式 | +2 | 高 | 明确的风险信号 |
| 回头客 | -1 | 低 | 历史记录良好 |
| 高信用 | -1 | 低 | 信用评级良好 |
| 定期操作 | -1 | 低 | 可预期的正常行为 |
| 白名单 | -2 | 高 | 已验证的高信任用户 |

## 3. 动态评分算法

### 3.1 评分公式

```
┌─────────────────────────────────────────────────────────────┐
│                    动态风险评分                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  最终风险分 = 基础等级分 + Σ加分因子 - Σ减分因子           │
│                                                             │
│  基础等级分：                                               │
│  ├── L0（查询）：0分                                      │
│  ├── L1（<1万）：1分                                      │
│  └── L2（>1万）：2分                                      │
│                                                             │
│  最终等级 = clamp(最终风险分, L0, L2)                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 评分示例

```
示例1：老客户白天操作1万订单
- 基础分：L1 = 1分
- 回头客：-1分
- 高信用：-1分
- 最终分：1 - 1 - 1 = -1 → 降为L0

示例2：新客户凌晨操作15万订单
- 基础分：L2 = 2分
- 新客户：+1分
- 非工作时间：+1分
- 最终分：2 + 1 + 1 = 4 → 升为L2（即使原金额<10万）

示例3：白名单客户操作100万
- 基础分：L2 = 2分
- 白名单：-2分
- 最终分：2 - 2 = 0 → 降为L0
```

## 4. 升降级规则

### 4.1 自动升级

```
触发条件：
├── 单笔金额 > 当前等级阈值2倍
├── 24小时内同一类型操作 > 10次
├── 账户出现异常登录
├── 客户信用评级下降
└── 检测到攻击模式

升级处理：
- 自动升一级（L0→L1, L1→L2）
- 加强监控
- 触发告警
```

### 4.2 自动降级

```
触发条件：
├── 连续10次操作无否决
├── 月交易额 > 阈值且通过率>99%
└── 客户评级提升

降级处理：
- 自动降一级
- 享受更快的处理速度
- 仍需监控
```

### 4.3 升降级限制

```
限制规则：
- 最多连续升级2次
- 最多连续降级3次
- L0不能再降级
- L2不能再升级
```

## 5. 实时风险信号

### 5.1 风险信号类型

| 信号类型 | 说明 | 权重 |
|----------|------|------|
| **强信号** | 明确的风险指示 | +2~3 |
| **中信号** | 潜在风险指示 | +1~2 |
| **弱信号** | 需要关注 | +0~1 |

### 5.2 强信号示例

```
强信号（立即处理）：
- 账户在多个地点同时登录
- 短时间大量请求
- 检测到自动化脚本
- 违反风控规则
- 被其他系统标记
```

### 5.3 中信号示例

```
中信号（加强监控）：
- 操作频率突然变化
- 金额分布异常
- 非惯常操作时间
- 操作设备变化
- 操作地域变化
```

## 6. 动态阈值机制

### 6.1 金额阈值动态调整

```
┌─────────────────────────────────────────────────────────────┐
│                    动态金额阈值                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  基础阈值：                                                 │
│  ├── L0：query，无金额                                     │
│  ├── L1：<1万                                             │
│  └── L2：>1万                                             │
│                                                             │
│  动态调整：                                                 │
│  ├── 客户月交易>50万 → 阈值提升20%                        │
│  ├── 客户月交易>100万 → 阈值提升50%                       │
│  ├── 新客户 → 阈值降低50%                                 │
│  └── 高风险客户 → 阈值降低50%                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 时间因子

```
非工作时间权重：
- 工作日9:00-18:00：标准权重
- 工作日18:00-22:00：+0.5权重
- 工作日22:00-09:00：+1权重
- 周末/节假日：+0.5权重
```

## 7. 机器学习辅助（可选）

### 7.1 异常检测模型

```
使用历史数据训练异常检测模型：
- 输入：用户行为特征
- 输出：异常分数
- 应用：辅助动态风险评分

优势：
- 发现规则难以覆盖的异常
- 适应用户行为变化
- 减少人工规则维护
```

### 7.2 实施建议

```
MVP阶段：
- 仅使用规则系统
- 积累标注数据

成长阶段：
- 引入无监督异常检测
- 用历史数据验证效果

成熟阶段：
- 引入有监督风险评分模型
- 持续迭代优化
```

## 8. 监控与告警

### 8.1 监控指标

| 指标 | 说明 | 正常范围 |
|------|------|----------|
| 升级率 | 被升级的操作比例 | <10% |
| 降级率 | 被降级的操作比例 | >30% |
| 信号命中率 | 强信号触发的比例 | <1% |
| 动态评分准确率 | 动态评分后实际风险比例 | >90% |

### 8.2 告警规则

```markdown
┌─────────────────────────────────────────────────────────────┐
│                    告警规则                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  实时告警（立即处理）：                                     │
```

## 9. 实施建议

### 9.1 MVP阶段简化

```
MVP阶段建议：
├── 仅使用基础因子（操作类型+金额）
├── 加减分因子从简（最多3-5个）
├── 不引入机器学习
└── 阈值固定，不做动态调整

这样做的好处：
1. 快速上线
2. 问题易排查
3. 用户容易理解
```

### 9.2 渐进增强路径

```
Phase 1 (MVP)：
- 规则系统
- 基础因子
- 固定阈值

Phase 2：
- 增加加减分因子
- 实现自动升降级

Phase 3：
- 引入时间因子
- 动态阈值调整

Phase 4：
- 引入机器学习
- 异常检测
```
